## Overview

Proper configuration is critical for a secure and performant production deployment. This guide covers essential configuration settings for Quejas Ciudadanas in a production environment.

<Warning>
  Never commit sensitive configuration values (passwords, connection strings) to source control.
</Warning>

## Application Configuration Files

### Configuration File Hierarchy

The application uses the standard ASP.NET Core configuration system with the following hierarchy:

1. **appsettings.json** - Base configuration (committed to source control)
2. **appsettings.Production.json** - Production-specific overrides
3. **Environment Variables** - Server-specific sensitive values
4. **Azure Key Vault / Secrets Manager** - Enterprise secrets management (optional)

<Note>
  Settings in later sources override earlier ones. Use this hierarchy to separate public and sensitive configuration.
</Note>

## Database Configuration

### Connection String Setup

The application requires a SQL Server connection string.

<Steps>
  <Step title="Review Current Configuration">
    Current `appsettings.json` structure:
    
    ```json
    {
      "ConnectionStrings": {
        "QuejasCiudadanaDb": "Server=localhost;Database=quejas_ciudadana;..."
      }
    }
    ```
  </Step>
  
  <Step title="Create Production Configuration">
    Create `appsettings.Production.json` in the application root:
    
    ```json
    {
      "ConnectionStrings": {
        "QuejasCiudadanaDb": "Server=prod-sql-server;Database=quejas_ciudadana_prod;User Id=app_user;Password=PLACEHOLDER;Encrypt=True;TrustServerCertificate=False;MultipleActiveResultSets=True;"
      },
      "Logging": {
        "LogLevel": {
          "Default": "Warning",
          "Microsoft.AspNetCore": "Warning",
          "Microsoft.EntityFrameworkCore": "Error"
        }
      },
      "AllowedHosts": "yourdomain.com,www.yourdomain.com"
    }
    ```
  </Step>
</Steps>

### Connection String Security Options

#### Option 1: Environment Variables (Recommended)

Store the connection string in an environment variable:

<Steps>
  <Step title="Set Environment Variable">
    On the server, set the connection string as a system environment variable:
    
    ```powershell
    [System.Environment]::SetEnvironmentVariable(
        'ConnectionStrings__QuejasCiudadanaDb',
        'Server=prod-server;Database=quejas_ciudadana_prod;User Id=app_user;Password=SecurePassword123!;Encrypt=True;TrustServerCertificate=False;MultipleActiveResultSets=True;',
        [System.EnvironmentVariableTarget]::Machine
    )
    ```
    
    <Note>
      The double underscore `__` syntax represents nested configuration keys in environment variables.
    </Note>
  </Step>
  
  <Step title="Restart IIS">
    After setting environment variables, restart IIS to apply changes:
    
    ```powershell
    net stop was /y
    net start w3svc
    ```
  </Step>
  
  <Step title="Remove from Configuration File">
    In `appsettings.Production.json`, you can now use a placeholder:
    
    ```json
    {
      "ConnectionStrings": {
        "QuejasCiudadanaDb": "SET_VIA_ENVIRONMENT_VARIABLE"
      }
    }
    ```
  </Step>
</Steps>

#### Option 2: Encrypted Configuration Sections

<Warning>
  Windows-based configuration encryption using DPAPI is available but complex. Environment variables are typically sufficient for most deployments.
</Warning>

### Connection String Parameters Explained

```plaintext
Server=prod-sql-server          # SQL Server hostname or IP
Database=quejas_ciudadana_prod  # Database name
User Id=app_user                # SQL authentication username
Password=SecurePassword123!     # SQL authentication password
Encrypt=True                    # Enable TLS encryption (required for Azure SQL)
TrustServerCertificate=False    # Validate SSL certificate (False for production)
MultipleActiveResultSets=True   # Allow multiple active result sets
```

<Tip>
  For Azure SQL Database, use:
  ```
  Server=tcp:yourserver.database.windows.net,1433;
  Database=quejas_ciudadana;
  User Id=username@yourserver;
  Password=password;
  Encrypt=True;
  TrustServerCertificate=False;
  Connection Timeout=30;
  ```
</Tip>

## Security Configuration

### Session Security

The application uses session management for authentication. Ensure secure session configuration:

```csharp
// In Program.cs
builder.Services.AddSession(options =>
{
    options.IdleTimeout = TimeSpan.FromMinutes(30);
    options.Cookie.HttpOnly = true;          // Prevent JavaScript access
    options.Cookie.IsEssential = true;       // Required for GDPR compliance
    options.Cookie.SecurePolicy = CookieSecurePolicy.Always;  // HTTPS only
    options.Cookie.SameSite = SameSiteMode.Strict;  // CSRF protection
});
```

<Warning>
  Verify these settings in your deployed `Program.cs`. The application should enforce HTTPS-only cookies in production.
</Warning>

### HTTPS Enforcement

Ensure HTTPS is enforced at the application level:

```csharp
// In Program.cs - add before UseStaticFiles()
if (!app.Environment.IsDevelopment())
{
    app.UseHsts();  // Enable HTTP Strict Transport Security
}

app.UseHttpsRedirection();  // Redirect HTTP to HTTPS
```

<Steps>
  <Step title="Enable HSTS">
    HTTP Strict Transport Security (HSTS) forces browsers to use HTTPS.
    
    Add to `Program.cs`:
    ```csharp
    builder.Services.AddHsts(options =>
    {
        options.Preload = true;
        options.IncludeSubDomains = true;
        options.MaxAge = TimeSpan.FromDays(365);
    });
    ```
  </Step>
  
  <Step title="Configure HSTS in web.config">
    Add to `web.config` in `<system.webServer>`:
    
    ```xml
    <httpProtocol>
      <customHeaders>
        <add name="Strict-Transport-Security" 
             value="max-age=31536000; includeSubDomains" />
      </customHeaders>
    </httpProtocol>
    ```
  </Step>
</Steps>

### Security Headers

Add security headers to protect against common web vulnerabilities:

```xml
<!-- In web.config -->
<system.webServer>
  <httpProtocol>
    <customHeaders>
      <add name="X-Content-Type-Options" value="nosniff" />
      <add name="X-Frame-Options" value="SAMEORIGIN" />
      <add name="X-XSS-Protection" value="1; mode=block" />
      <add name="Referrer-Policy" value="strict-origin-when-cross-origin" />
      <add name="Permissions-Policy" value="geolocation=(self), microphone=(), camera=()" />
    </customHeaders>
  </httpProtocol>
</system.webServer>
```

## Logging Configuration

### Production Logging Levels

Configure appropriate logging levels for production in `appsettings.Production.json`:

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore.Database.Command": "Error",
      "Microsoft.Hosting.Lifetime": "Information"
    }
  }
}
```

<Note>
  **Recommended Levels:**
  - **Information**: Application startup/shutdown only
  - **Warning**: Recoverable issues
  - **Error**: Unhandled exceptions and errors
  - Avoid **Debug** or **Trace** in production (performance impact)
</Note>

### File Logging (Optional)

For persistent logging, consider adding a file logging provider:

<Steps>
  <Step title="Install Serilog (Optional)">
    ```bash
    dotnet add package Serilog.AspNetCore
    dotnet add package Serilog.Sinks.File
    ```
  </Step>
  
  <Step title="Configure Serilog">
    In `Program.cs`:
    
    ```csharp
    using Serilog;
    
    Log.Logger = new LoggerConfiguration()
        .WriteTo.File(
            "logs/quejas-ciudadanas-.txt",
            rollingInterval: RollingInterval.Day,
            retainedFileCountLimit: 30
        )
        .CreateLogger();
    
    builder.Host.UseSerilog();
    ```
  </Step>
  
  <Step title="Create Logs Directory">
    ```powershell
    New-Item -Path "C:\inetpub\quejas_ciudadanas\logs" -ItemType Directory
    ```
    
    Grant write permissions to the application pool identity.
  </Step>
</Steps>

### Windows Event Log

Alternatively, log to Windows Event Log:

```csharp
builder.Logging.AddEventLog(settings =>
{
    settings.SourceName = "QuejasCiudadanas";
    settings.LogName = "Application";
});
```

## Performance Configuration

### Response Compression

Enable response compression for better performance:

```csharp
builder.Services.AddResponseCompression(options =>
{
    options.EnableForHttps = true;
});

var app = builder.Build();
app.UseResponseCompression();
```

### Database Connection Pooling

SQL Server connection pooling is enabled by default. Optimize with:

```plaintext
Min Pool Size=5;
Max Pool Size=100;
Pooling=true;
```

### Application Pool Settings

Optimize IIS Application Pool settings:

<AccordionGroup>
  <Accordion title="Recycling Settings">
    Configure in IIS Manager → Application Pool → Recycling:
    
    - **Regular Time Interval**: 1740 minutes (29 hours)
    - **Specific Times**: Off (or schedule during low-traffic periods)
    - **Memory-based recycling**:
      - Virtual Memory Limit: 0 (unlimited)
      - Private Memory Limit: 2097152 KB (2 GB) - adjust based on server RAM
  </Accordion>
  
  <Accordion title="Process Model">
    - **Idle Time-out**: 20 minutes (or 0 for high-traffic sites)
    - **Maximum Worker Processes**: 1 (increase for web garden if needed)
    - **Ping Enabled**: True
    - **Ping Maximum Response Time**: 90 seconds
  </Accordion>
  
  <Accordion title="Rapid-Fail Protection">
    - **Enabled**: True
    - **Maximum Failures**: 5
    - **Failure Interval**: 5 minutes
  </Accordion>
</AccordionGroup>

## File Upload Configuration

### Upload Limits

Configure maximum file upload sizes:

```xml
<!-- In web.config -->
<system.webServer>
  <security>
    <requestFiltering>
      <!-- 50 MB limit -->
      <requestLimits maxAllowedContentLength="52428800" />
    </requestFiltering>
  </security>
</system.webServer>
```

And in `Program.cs`:

```csharp
builder.Services.Configure<FormOptions>(options =>
{
    options.MultipartBodyLengthLimit = 52428800; // 50 MB
});
```

### Upload Directory Permissions

<Warning>
  Ensure the upload directory has appropriate permissions and is NOT within `wwwroot` to prevent direct execution of uploaded files.
</Warning>

```powershell
# Create upload directory outside wwwroot
New-Item -Path "C:\AppData\QuejasCiudadanas\Uploads" -ItemType Directory

# Grant permissions
$acl = Get-Acl "C:\AppData\QuejasCiudadanas\Uploads"
$rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "IIS AppPool\QuejasCiudadanasAppPool",
    "Modify",
    "ContainerInherit,ObjectInherit",
    "None",
    "Allow"
)
$acl.SetAccessRule($rule)
Set-Acl "C:\AppData\QuejasCiudadanas\Uploads" $acl
```

## Monitoring and Health Checks

### Application Insights (Azure)

For Azure deployments, integrate Application Insights:

```bash
dotnet add package Microsoft.ApplicationInsights.AspNetCore
```

```csharp
builder.Services.AddApplicationInsightsTelemetry();
```

### Health Check Endpoint

Implement a health check endpoint:

```csharp
builder.Services.AddHealthChecks()
    .AddSqlServer(connectionString);

app.MapHealthChecks("/health");
```

Access at: `https://yourdomain.com/health`

## Backup and Recovery

<Steps>
  <Step title="Database Backups">
    Configure SQL Server automated backups:
    - Full backup: Daily
    - Differential backup: Every 6 hours
    - Transaction log backup: Every 15 minutes
  </Step>
  
  <Step title="Application Files">
    Backup the application directory and configuration:
    - `C:\inetpub\quejas_ciudadanas`
    - `web.config`
    - `appsettings.Production.json`
  </Step>
  
  <Step title="Uploaded Content">
    Regularly backup the uploads directory:
    - Schedule using Windows Task Scheduler
    - Consider cloud backup solutions (Azure Blob Storage, AWS S3)
  </Step>
</Steps>

## Configuration Checklist

Before going live, verify:

- [ ] Connection string stored securely (environment variable or encrypted)
- [ ] `ASPNETCORE_ENVIRONMENT` set to `Production`
- [ ] HTTPS enforced with valid SSL certificate
- [ ] Logging level set to Warning or Error
- [ ] Session cookies configured with HttpOnly and Secure flags
- [ ] Security headers configured in web.config
- [ ] File upload limits configured appropriately
- [ ] Upload directory permissions set correctly
- [ ] Database backups scheduled and tested
- [ ] Health check endpoint functional
- [ ] Application pool recycling configured
- [ ] Error handling tested
- [ ] Monitoring/logging solution in place
