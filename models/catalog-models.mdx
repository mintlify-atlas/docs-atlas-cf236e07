## Overview

Catalog models provide reference data used throughout the application. These models define dropdown options, problem taxonomies, geographic locations, and system states.

## User Catalogs

### TipoIdentificacion

Types of identification documents.

```csharp
public class TipoIdentificacion
{
    [Key]
    public int Id_identificacion { get; set; }
    public string Descripcion { get; set; }
}
```

#### Properties

| Property | Type | Description |
|----------|------|-------------|
| `Id_identificacion` | `int` | Primary key - Unique identifier |
| `Descripcion` | `string` | Document type name (e.g., "Cédula", "Pasaporte") |

## Problem Taxonomy

The system uses a three-level hierarchy for categorizing problems:

1. **Problema** (Problem Category) - Top level (e.g., "Infrastructure")
2. **Clase_Problema** (Problem Class) - Mid level (e.g., "Roads")
3. **Tipo_Problema** (Problem Type) - Specific type (e.g., "Pothole")

### Problema

Top-level problem categories.

```csharp
public class Problema
{
    [Key]
    public int Id_Problema { get; set; }
    public string Nombre { get; set; }
    public ICollection<Clase_Problema> Clases_Problema { get; set; }
}
```

#### Properties

| Property | Type | Description |
|----------|------|-------------|
| `Id_Problema` | `int` | Primary key - Problem category ID |
| `Nombre` | `string` | Category name |
| `Clases_Problema` | `ICollection<Clase_Problema>` | Navigation property to related problem classes |

### Clase_Problema

Mid-level problem classification.

```csharp
public class Clase_Problema
{
    [Key]
    public int Id_Clase_Problema { get; set; }
    public string Nombre { get; set; }
    public int Id_Problema { get; set; }
    public Problema Problema { get; set; }
    public ICollection<Tipo_Problema> Tipos_Problema { get; set; }
}
```

#### Properties

| Property | Type | Description |
|----------|------|-------------|
| `Id_Clase_Problema` | `int` | Primary key - Problem class ID |
| `Nombre` | `string` | Class name |
| `Id_Problema` | `int` | Foreign key to parent Problema |
| `Problema` | `Problema` | Navigation property to parent problem |
| `Tipos_Problema` | `ICollection<Tipo_Problema>` | Navigation property to related problem types |

### Tipo_Problema

Specific problem types.

```csharp
public class Tipo_Problema
{
    [Key]
    public int Id_Tipo_Problema { get; set; }
    public string Nombre { get; set; }
    public int Id_Clase_Problema { get; set; }
    public Clase_Problema Clase_Problema { get; set; }
}
```

#### Properties

| Property | Type | Description |
|----------|------|-------------|
| `Id_Tipo_Problema` | `int` | Primary key - Problem type ID |
| `Nombre` | `string` | Type name |
| `Id_Clase_Problema` | `int` | Foreign key to parent Clase_Problema |
| `Clase_Problema` | `Clase_Problema` | Navigation property to parent problem class |

## Relationships Overview

### Problem Hierarchy

```
Problema (Infrastructure)
└── Clase_Problema (Roads)
    └── Tipo_Problema (Pothole)
    └── Tipo_Problema (Damaged Sidewalk)
└── Clase_Problema (Public Buildings)
    └── Tipo_Problema (Damaged Roof)
```

### Geographic Hierarchy

```
CAT001_DEPTOS (Department)
└── CAT002_CIUDADES (City 1)
└── CAT002_CIUDADES (City 2)
└── CAT002_CIUDADES (City 3)
```

## Usage in Forms

Catalog models are typically used to populate dropdown lists:

- User registration forms use `TipoIdentificacion` and `Sexo`
- Report creation forms use the problem taxonomy (`Problema` → `Clase_Problema` → `Tipo_Problema`)
- Location selectors use `CAT001_DEPTOS` and `CAT002_CIUDADES`
- Status filters use `Estado_caso`

## Database Relationships

All catalog models use Entity Framework navigation properties to establish relationships:

- Parent-child relationships (one-to-many)
- Foreign key constraints
- Lazy loading support through `ICollection<T>` properties
